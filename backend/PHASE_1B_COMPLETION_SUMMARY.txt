================================================================================
API ENDPOINTS
================================================================================

GET /api/config/admin
  Authentication: Required (JWT + Super Admin role)
  Response: IAdminConfig
  Use Case: Admin panel settings page
  
GET /api/config/mobile
  Authentication: None (device-based identity)
  Response: IMobileConfig
  Use Case: Mobile app checks rate limits before submitting
  
GET /api/config
  Authentication: Required (JWT + Super Admin role)
  Response: ConfigEntry[] (all config)
  Use Case: Admin panel configuration management (backward compat)
  
PATCH /api/config/:key
  Authentication: Required (JWT + Super Admin role)
  Body: { value: string }
  Response: Updated ConfigEntry
  Use Case: Admin updates specific config value

================================================================================
DESIGN PRINCIPLES
================================================================================

✅ Interface Segregation Principle (ISP)
   - Each domain has its own interface
   - Services only depend on what they need
   - No monolithic GlobalConfigService exposure

✅ Dependency Inversion Principle (DIP)
   - Services depend on abstractions (interfaces)
   - Easier to test (mock only needed methods)
   - Clearer contracts

✅ Single Responsibility Principle (SRP)
   - GlobalConfigService: persistence + cache
   - Providers: domain-specific retrieval
   - Controller: HTTP routing + authorization

✅ Least Privilege (Security)
   - Mobile never sees session_timeout_minutes
   - Automation never sees admin settings
   - Each consumer sees ONLY what it needs
